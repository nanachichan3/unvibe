## 1. Project Overview

**Unvibe** is an AI-powered code reading game platform. Users upload a codebase (zip archive) or connect a GitHub repository, and the platform generates quiz-style games that test and teach code comprehension. Every game is a code reading exercise — players look at real code from their own codebase and answer questions about it.

**Product stage:** MVP (per content-calendar.md: "4-Week Plan, MVP Stage")  
**Current game types implemented:** 11 game types (6 core parser-based + 5 GitHub-powered)  
**Deployment:** Docker, standalone Next.js output

- **Gemini API** (via REST, direct `generateContent` + `countTokens` endpoints)
- Model options: `gemini-2.5-flash` (default), `gemini-2.0-flash-exp`, `gemini-1.5-flash`, `gemini-1.5-flash-8b`
- Token counting via `countTokens` endpoint with localStorage persistence
- No OpenAI support currently (AI client configured for Gemini only)

### External APIs
- **GitHub REST API v3** — repo contents, commit history, blame data, contributor info
- **Google Generative Language API** — Gemini AI

### Security
- Zip slip protection via `sanitizeZipPath()`
- Zip bomb detection (compression ratio guard + total uncompressed size cap)
- Per-file size limits (100MB max) and archive size limits (500MB max)
- 50,000 file count cap

## 4. Key Architectural Decisions

### Client-Side Heavy Architecture
The core parsing and metric computation runs entirely **client-side** in the browser. The server only serves the Next.js static/standalone build. This is a deliberate choice:
- Zero backend infrastructure for MVP
- No file upload endpoint needed — zip parsing happens in-browser via JSZip
- Privacy: user code never leaves the browser (except when using GitHub mode)

### Seeded PRNG for Reproducibility
`generateRoundQuestion()` uses **mulberry32** (a seeded PRNG) so that the same `(roundKey, gameType, seed)` always produces the same question. This makes sessions reproducible and debuggable.

### Two-Mode Game Design (Simple vs AI)
Every game has two modes:
- **Simple:** Parser/heuristic-only, no AI, instant, free
- **AI:** LLM generates content (distractors, analysis); token cost shown before generation

### Architecture Phases (from REDESIGN.md)
1. **Phase 1:** Core parser + Games 1, 2, 4 ✅ (implemented)
2. **Phase 2:** Games 3, 5, 6, 7 ✅ (implemented)
3. **Phase 3:** GitHub Games 8, 9, 10, 11 ✅ (implemented)
4. **Phase 4:** Polish — hints, streaks, VIM themes (not yet started)

### Mode Toggle + Token Counter
`ModeToggle` and `TokenMeter` are persistent UI elements. Session token counts accumulate across rounds and are stored in `localStorage` via `getStoredModel()`. Token counting uses Gemini's official `countTokens` endpoint with a rough fallback.

### GitHub Integration
GitHub mode fetches repository data via GitHub REST API v3. Supports:
- Public repos via unauthenticated requests (rate limited)
- Authenticated requests via personal access token (higher limits)
- Recursive tree fetch via `git/trees` API for fast directory listing
- Commit history per file for timeline games
- Blame-like data (uses commits API as proxy)
